# Feature Specification: Enhanced Student Dashboard & Module Progress Detail

**Feature Branch**: `017-enhanced-dashboard`
**Created**: 2026-05-17
**Status**: Draft
**Input**: User description: "Enhanced Student Dashboard with Module Progress Detail — add charts, AI recommendations (background task), refine module cards with View Progress, and a per-module progress detail page showing covered/remaining topics. Remove standalone /progress page and sidebar link."

---

## Overview

The student dashboard (`/dashboard`) becomes the single destination for all progress visibility. It gains richer data visualisations and AI-generated recommendations delivered asynchronously so the page is never blocked. Curriculum module cards are refined and their primary action changes to "View Progress", opening a dedicated per-module detail page that shows which topics the student has covered and which remain — also generated asynchronously by an agent.

The standalone `/progress` page and its sidebar navigation link are removed entirely; all their value is absorbed into the enhanced dashboard and the new module detail page.

---

## Clarifications

### Session 2026-05-18

- Q: Which async delivery mechanism should be used for the Recommended Next card and the module progress detail page? → A: SSE (Server-Sent Events), reusing the existing F15 `/stream` endpoint pattern.
- Q: Which charting library should be used for the dashboard charts? → A: Recharts (React-native SVG, declarative, compatible with Next.js 14+).
- Q: When multiple mastery records exist for the same calendar day, how should the "Mastery Over Time" chart aggregate them? → A: Average — compute mean mastery percentage across all records for that day.
- Q: What is the format of the `moduleId` URL segment in `/module/[moduleId]/progress`? → A: Human-readable slug (e.g., `control-flow`, `data-structures`).
- Q: Where does the module topic metadata live that the Progress Agent uses to produce topic-level breakdowns? → A: Hardcoded in the Progress Agent's system prompt as static topic lists per module.
- Q: Should SSE delivery reuse the existing chat stream endpoint (extended with a mode parameter) or use new dedicated endpoints? → A: New dedicated endpoints — `GET /api/v1/dashboard/recommendations/stream` and `GET /api/v1/module/{moduleId}/progress/stream`.
- Q: What "activity history" data should the backend pass to the Progress Agent for module topic breakdown and recommendations? → A: Mastery scores only (all modules from `mastery_records`).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Student sees richer dashboard with charts (Priority: P1)

A student visits their dashboard and immediately sees two charts alongside the existing module cards and streak counter: a line chart showing their overall mastery score trend over time, and a horizontal bar chart comparing mastery levels across all eight Python modules (colour-coded by mastery band).

**Why this priority**: Charts are the most universally visible improvement and deliver value without any agent dependency — they render from data already stored in the database.

**Independent Test**: Navigate to `/dashboard` as a student with at least one past mastery record. Both charts render with correct data. Page loads without waiting for any agent response.

**Acceptance Scenarios**:

1. **Given** a student has completed at least one module activity, **When** they visit `/dashboard`, **Then** a line chart titled "Mastery Over Time" shows their overall mastery percentage plotted across calendar dates.
2. **Given** a student with mixed mastery across modules, **When** they view the dashboard, **Then** a horizontal bar chart shows one bar per module (all 8), colour-coded: Red (0–40%), Yellow (41–70%), Green (71–90%), Blue (91–100%).
3. **Given** a brand-new student with no activity, **When** they visit `/dashboard`, **Then** both charts render with an empty-state message ("Start learning to see your progress here") rather than a blank or broken chart.

---

### User Story 2 — Student receives AI-generated recommendations (Priority: P1)

A student visits their dashboard and sees a "Recommended Next" card. The recommendations are generated by the Progress Agent in the background. While the agent is working, the card shows a friendly placeholder. Once complete, the card updates with personalised suggestions (e.g., "You're close to proficiency in Control Flow — take a quiz to push past 70%").

**Why this priority**: Recommendations drive what the student does next — they directly affect engagement and learning outcomes. Non-blocking delivery means the rest of the dashboard is never held hostage.

**Independent Test**: Load the dashboard. The "Recommended Next" card appears immediately with a loading state. Within a reasonable time it updates with AI content, without any page reload.

**Acceptance Scenarios**:

1. **Given** a student loads the dashboard, **When** the Progress Agent has not yet responded, **Then** the "Recommended Next" card displays a friendly loading message (e.g., "Personalising your recommendations…") and the rest of the dashboard is fully interactive.
2. **Given** the Progress Agent completes its analysis, **When** the recommendation is ready, **Then** the card updates in-place (no page reload) with 1–3 specific, actionable recommendations tailored to the student's weakest modules and recent activity.
3. **Given** the Progress Agent fails or times out after 30 seconds, **When** no recommendation is returned, **Then** the card shows a graceful fallback message ("We couldn't load recommendations right now — try refreshing") and does not display an error stack or blank space.

---

### User Story 3 — Student navigates from module card to module progress detail (Priority: P1)

The curriculum module cards on the dashboard now show a "View Progress" button instead of "Start Lesson". Clicking it navigates the student to `/module/[moduleId]/progress`, a dedicated page showing exactly which topics within that module they have covered and which remain.

**Why this priority**: This is the primary navigation change — it replaces the removed `/progress` page with a focused, module-level view that is more actionable for the student.

**Independent Test**: From the dashboard, click "View Progress" on any module card. The module detail page loads and shows at minimum a list of topics with covered/remaining status.

**Acceptance Scenarios**:

1. **Given** a student is on the dashboard, **When** they click "View Progress" on the "Control Flow" module card, **Then** they are navigated to `/module/control-flow/progress`.
2. **Given** a student is on the module progress page, **When** the page loads, **Then** it shows a layout distinguishing covered topics (e.g., "if/else ✓") from remaining topics (e.g., "ternary shorthand ✗").
3. **Given** a student visits the page for a module they have never touched, **When** the page loads, **Then** all topics show as "Not yet started" and a call-to-action encourages them to begin.

---

### User Story 4 — Module progress detail is generated asynchronously by an agent (Priority: P2)

The topic breakdown on `/module/[moduleId]/progress` is generated by the Progress Agent. The page renders immediately with a skeleton/loading state while the agent works. Once the analysis arrives, it replaces the skeleton without a page reload.

**Why this priority**: Agent generation enables nuanced topic-level insight (e.g., "knows basic for loops but not nested loops") that cannot come from simple database queries. Non-blocking delivery keeps the page responsive.

**Independent Test**: Visit a module progress page. A loading skeleton appears instantly. Within a reasonable time it is replaced by the agent-generated topic breakdown, without navigating away or refreshing.

**Acceptance Scenarios**:

1. **Given** a student navigates to `/module/[moduleId]/progress`, **When** the page first renders, **Then** a loading skeleton is shown in the topic list area while the agent generates the breakdown.
2. **Given** the Progress Agent returns a topic analysis, **When** the data arrives, **Then** the skeleton is replaced with the full topic breakdown showing covered topics, remaining topics, and any partial-progress notes (e.g., "Understands basic syntax but not edge cases").
3. **Given** the Progress Agent fails or times out after 30 seconds, **When** no data is returned, **Then** the page shows a graceful fallback ("We couldn't analyse your progress right now — try again shortly") with a retry button.

---

### User Story 5 — Standalone /progress page and sidebar link are removed (Priority: P1)

The `/progress` page no longer exists. The sidebar navigation link pointing to it is removed. Any direct URL access returns a 404 or redirects to the dashboard.

**Why this priority**: Removing the page is a prerequisite for ensuring students use the new, richer module-level experience and do not land on a dead or empty page.

**Independent Test**: Navigate directly to `/progress`. Receive a 404 or redirect. Confirm sidebar has no "Progress" link.

**Acceptance Scenarios**:

1. **Given** a student navigates to `/progress`, **When** the page loads, **Then** they see a 404 page or are redirected to `/dashboard` — not a blank or broken page.
2. **Given** a student looks at the sidebar navigation, **When** they scan all links, **Then** no "Progress" link exists.

---

### Edge Cases

- What happens if a student has mastery records but no timestamped history? The "Mastery Over Time" chart shows a single data point with an empty-state hint to keep learning.
- What if the module has no defined topics in the database? The module detail page shows only the agent-generated topic list as the source of truth.
- What if the agent returns topics in a different order each time? Topics are sorted consistently: covered first, then remaining, alphabetically within each group.
- What if the student's screen is narrow (mobile)? Charts stack vertically; the module detail list remains scrollable.
- What if the agent returns an empty recommendations list? The "Recommended Next" card shows "You're all caught up! Keep practising to improve your scores."
- What if the student opens two tabs to the same module progress page? Each tab triggers its own independent agent call; there is no shared state between tabs.

## Requirements *(mandatory)*

### Functional Requirements

**Dashboard Charts**

- **FR-001**: The dashboard MUST display a line chart ("Mastery Over Time") showing the student's overall mastery percentage plotted by date, one data point per day with recorded activity. When multiple mastery records exist for the same day, the data point MUST use the **average** mastery percentage across all that day's records. Implemented using **Recharts** `<LineChart>`.
- **FR-002**: The dashboard MUST display a horizontal bar chart ("Module Mastery") showing the mastery percentage for each of the 8 Python modules, colour-coded by mastery band (Red 0–40% / Yellow 41–70% / Green 71–90% / Blue 91–100%). Implemented using **Recharts** `<BarChart layout="vertical">`.
- **FR-003**: Both charts MUST render an empty-state message when the student has no historical mastery data, rather than a blank or broken chart area.

**AI Recommendations — Background Task**

- **FR-004**: The "Recommended Next" card MUST appear on the dashboard immediately on page load, showing a friendly loading state while the Progress Agent processes.
- **FR-005**: The Progress Agent call MUST be non-blocking — no other dashboard element may wait for it before rendering.
- **FR-006**: When the Progress Agent returns, the card MUST update in-place (no full page reload) with 1–3 actionable recommendations.
- **FR-007**: If the Progress Agent does not return within 30 seconds or returns an error, the card MUST show a graceful fallback message with a retry option.

**Curriculum Module Cards**

- **FR-008**: Each module card on the dashboard MUST replace the "Start Lesson" primary action with a "View Progress" button.
- **FR-009**: Clicking "View Progress" on a module card MUST navigate the student to `/module/[moduleId]/progress`, where `moduleId` is the module's human-readable slug (e.g., `control-flow`, `data-structures`).

**Module Progress Detail Page**

- **FR-010**: The `/module/[moduleId]/progress` page MUST render a loading skeleton in the topic list area immediately on load — before the agent responds.
- **FR-011**: The Progress Agent MUST be invoked as a non-blocking background task, providing the module ID, the authenticated student's ID, and the student's mastery scores across all modules (fetched from `mastery_records`) as context. No code submissions or quiz attempt data are included.
- **FR-012**: The agent-generated topic breakdown MUST categorise each topic as one of: covered, partial, or remaining.
- **FR-013**: Topics MUST be displayed sorted: covered topics first (checkmark indicator), remaining/partial topics after; alphabetically within each group.
- **FR-014**: If the Progress Agent fails or times out (30 seconds), the page MUST show a graceful fallback with a retry button.
- **FR-015**: The page MUST be scoped strictly to the authenticated student — accessing `/module/[moduleId]/progress` MUST return only that student's data regardless of URL manipulation.

**Removed Elements**

- **FR-016**: The `/progress` standalone page MUST be removed; requests to that URL MUST return a 404 or redirect to `/dashboard`.
- **FR-017**: The sidebar navigation link to `/progress` MUST be removed.

---

### Key Entities

- **MasteryRecord**: A student's mastery score for a module at a point in time. Attributes: student ID, module ID, mastery percentage, recorded date. Used to populate both dashboard charts.
- **ModuleTopic**: A topic label within a module (e.g., "ternary shorthand" within "Control Flow"). Attributes: topic label, module ID, display order. Defined as module metadata; the agent maps student activity to these labels.
- **TopicProgressItem**: An agent-generated assessment of a single topic for a student. Attributes: topic label, status (covered / partial / remaining), optional detail note. Ephemeral — generated on demand, not persisted.
- **Recommendation**: An agent-generated suggestion for the student's next learning action. Attributes: text, associated module (optional). Ephemeral — generated on demand, not persisted.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: The dashboard reaches interactive state (all non-agent content visible and usable) within 2 seconds on a standard connection, regardless of Progress Agent response time.
- **SC-002**: Both charts render correct data for 100% of students who have at least one mastery record; the empty-state is shown for students with no records.
- **SC-003**: The "Recommended Next" card updates with agent-generated content within 15 seconds for 95% of requests under normal load.
- **SC-004**: The module progress detail page renders its loading skeleton within 1 second of navigation, and replaces it with agent content within 15 seconds for 95% of requests under normal load.
- **SC-005**: Zero instances of a student being able to access another student's module progress data via URL manipulation.
- **SC-006**: The `/progress` URL returns a non-200 response (404 or redirect) in 100% of cases after deployment.
- **SC-007**: Students can navigate from the dashboard to any module's progress detail page and back without a full browser page reload.

## Assumptions

- Timestamped mastery records already exist in the database from F12/F02 — the "Mastery Over Time" chart reads from these without requiring a new table.
- The 8 Python module topics are **hardcoded as static lists in the Progress Agent's system prompt**; the agent maps the student's activity history to these topic labels to produce the covered/remaining breakdown. No DB query is required for topic metadata.
- The "Recommended Next" card component already exists on the dashboard (F13) and can be updated to receive its content from an async agent call rather than static data.
- The Progress Agent (F12) can be extended with a new prompt/mode to produce topic-level breakdowns when given a module ID and student context.
- The async delivery mechanism for both the "Recommended Next" card and the module progress detail page is **Server-Sent Events (SSE)** via two new dedicated endpoints: `GET /api/v1/dashboard/recommendations/stream` and `GET /api/v1/module/{moduleId}/progress/stream`. These are stateless read-only endpoints, separate from the F15 chat stream, so chat quota and routing logic do not apply. The client opens an SSE connection, receives agent output incrementally, then closes the connection when the terminal event arrives.

---

## Out of Scope

- Teacher-facing views of individual student progress (covered by F19: Class Analytics).
- Exporting charts as images or CSV files.
- Push notifications when a student reaches a new mastery level.
- Manually editing or overriding mastery scores.
- Persistent storage of agent-generated topic breakdowns or recommendations (they are generated fresh on each visit).
- The "Start Lesson" flow itself — this feature only changes the button label and destination; lesson initiation is out of scope.
