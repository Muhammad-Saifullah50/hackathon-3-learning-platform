# ADR-0001: AI-Generated Dynamic Content Over Pre-Authored Curriculum

- **Status:** Accepted
- **Date:** 2026-05-11
- **Feature:** Platform-wide (affects F07–F12 Agent Layer, F13 Student Dashboard, F15 Chat Interface, F16 Quiz System)

## Context

The initial design of LearnFlow assumed a traditional e-learning model where curriculum authors write lesson content, exercises, and quiz questions in advance and store them in the database. F02 seeded 8 Python modules with structured `Lesson`, `Exercise`, and `Quiz` records containing pre-authored content.

During feature 15 (Chat Interface) planning, a fundamental product decision was made: the platform's AI agents are capable of generating pedagogically sound, personalized content on demand. Pre-authoring content introduces a bottleneck (content must be written before it can be taught), creates a static experience that cannot adapt to individual student context, and duplicates effort already handled by the agent layer.

The core user flow confirms this: when a student asks "How do for loops work?", the Concepts Agent generates an explanation tailored to that student's level — it does not retrieve a pre-written lesson. When a student requests practice, the Exercise Agent generates a problem calibrated to their current mastery score. Mastery is then updated based on performance on those generated exercises and quizzes.

The 8-module topic structure (Basics, Control Flow, Data Structures, Functions, OOP, Files, Errors, Libraries) is retained as a **navigation and progress framework** — it defines what topics exist and gives the dashboard something to track against. It is not a content library.

## Decision

**Content Model: Fully AI-Generated on Demand**

- **Curriculum structure**: The 8 Python modules and their topic labels are retained as a static scaffold — they define the learning map and anchor progress tracking (e.g., "Module 2: Loops — 60% complete"). No pre-authored lesson prose is stored or retrieved.
- **Concept explanations**: Generated by the Concepts Agent at request time, adapting language complexity, example choice, and depth to the student's mastery level and prior conversation context.
- **Exercises and coding challenges**: Generated by the Exercise Agent based on the student's current module, mastery score, and any struggle patterns. Stored in the database *after* generation for replay, grading history, and teacher review — not stored before use.
- **Quizzes**: Generated by the Exercise Agent on demand. Surfaced to the student as a card-based (flashcard-style) UI. Results are scored and fed into the mastery calculation formula.
- **Mastery calculation**: Unchanged — 40% exercises, 30% quizzes, 20% code quality, 10% streak — but all inputs come from AI-generated content interactions rather than a fixed content bank.
- **Guardrails**: All agent content generation is scoped to the student's current module/topic and kept within Python programming education. Off-topic generation is blocked at the Triage Agent level.

## Consequences

### Positive

- **Personalization at scale**: Every student receives explanations and exercises calibrated to their exact level, without manual content authoring per difficulty tier.
- **Eliminates content authoring bottleneck**: No curriculum writers needed to produce or update lesson text. Platform can launch and evolve without a content team.
- **Context-aware teaching**: Agents have full visibility into the student's code, errors, mastery scores, and chat history — pre-authored content cannot replicate this level of situational awareness.
- **Automatic curriculum freshness**: Python best practices and idioms in generated content reflect the LLM's training; no manual updates required when language conventions evolve.
- **Reduced database surface area**: The `Lesson` content fields become topic labels only. Pre-authoring tables (`Exercise`, `Quiz`) store generated outputs for history, not inputs for delivery.

### Negative

- **Determinism loss**: Two students asking the same question get different explanations. This makes QA harder — there is no canonical "correct lesson" to verify against.
- **LLM dependency at runtime**: Every concept explanation and exercise requires a live LLM call. If the LLM provider is unavailable, teaching is blocked (no static fallback content).
- **Token cost at scale**: Each student interaction consumes LLM tokens. High concurrency (many students studying simultaneously) increases inference costs significantly vs. serving cached pre-authored text.
- **Pedagogical consistency risk**: Without authored content, the quality and sequencing of explanations varies by LLM run. A student could receive an advanced explanation when they needed a basic one if the Concepts Agent miscalibrates.
- **Existing seeded data partially wasted**: F02 seeded lesson content records that now serve only as topic labels. The `content` field on the `Lesson` model becomes unused.

## Alternatives Considered

**Alternative A: Fully Pre-Authored Curriculum (Rejected)**
All lesson text, exercises, and quizzes written by human authors and stored in the database. Agents only provide tutoring on top of fixed content.
- Rejected because: Requires significant ongoing content authoring effort, produces a static experience that cannot adapt to individual student context, and duplicates work the agent layer already performs better.

**Alternative B: Hybrid — Pre-Authored Scaffold + AI Enrichment (Deferred)**
Core lesson outlines are authored; the Concepts Agent elaborates and adapts them at runtime. Exercises draw from a pre-authored bank but are selected and personalized by the Exercise Agent.
- Not selected for MVP because: Doubles the surface area (content authoring pipeline + agent pipeline), delays launch, and provides marginal benefit over pure generation. May be revisited if pedagogical consistency issues emerge in production.

**Alternative C: AI-Generated with Human Review Gate (Deferred)**
Content is AI-generated but passes through a teacher review step before being shown to students.
- Not selected for MVP because: Introduces a human bottleneck that breaks the real-time tutoring experience. Could be introduced later for teacher-assigned exercises (F21).
